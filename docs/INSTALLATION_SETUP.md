## Phase 1: Deploying Wazuh SIEM Manager (Host-A)

Wazuh is deployed on Host-A (Ubuntu 26.04) as a single-node containerized stack.

### 1. System Preparation & Virtual Memory Allocation
OpenSearch (Wazuh's database engine) requires high virtual memory limits. Run the following command to update limits and install Docker components:

```bash
# Update repositories and install Docker packages
sudo apt update && sudo apt install docker.io docker-compose-v2 -y
sudo usermod -aG docker $USER

# Configure virtual memory mappings
sudo sysctl -w vm.max_map_count=262144
echo "vm.max_map_count=262144" | sudo tee -a /etc/sysctl.conf
```
> [!NOTE]
> Log out of your server shell and log back in to apply Docker group privileges.

### 2. Clone Wazuh Repository & Generate Certs
We utilize the official Wazuh-Docker repository to construct our Single-Node manager:
```bash
# Clone repository to the /opt folder
cd /opt
sudo git clone https://github.com/wazuh/wazuh-docker.git
sudo chown -R $USER:$USER /opt/wazuh-docker
cd /opt/wazuh-docker

# Execute the cert generator container to create secure communication TLS keys
docker compose -f generate-indexer-certs.yml run --rm generator
```

### 3. Deploy Wazuh Stack
```bash
# Navigate to the single-node deployment directory and run containers in background
cd /opt/wazuh-docker/single-node
docker compose up -d
```

### 4. Access Dashboard
Wait 1–2 minutes for indexers to boot up. Open your browser and navigate to `https://<Host-A-IP>`.
* **Login Credentials:**
  * **Username:** `admin`
  * **Password:** `SecretPassword`

## Phase 2: Host-B Endpoint Isolation & SSH Port Migration

To avoid port conflicts (since Cowrie must intercept malicious traffic on port 22/23), Host-B's true administrative SSH service must be migrated to port `22022`.

### 1. Modify SSH Configuration
```bash
sudo nano /etc/ssh/sshd_config
```
Find the `#Port 22` or `Port 22` line and change it to:
```plaintext
Port 22022
```

### 2. Configure Systemd Socket Activation (Required for Ubuntu 22.04+)
Modern Ubuntu servers use socket-based ssh activation which overrides `sshd_config` configurations. You must edit the systemd socket:
```bash
sudo systemctl edit ssh.socket
```
Paste this configuration override at the bottom of the file:
```ini
[Socket]
ListenStream=
ListenStream=22022
```

### 3. Apply Settings and Configure UFW Firewall
```bash
# Configure system firewall
sudo ufw allow 22022/tcp
sudo ufw reload
sudo ufw enable

# Apply socket changes and restart SSH services
sudo systemctl daemon-reload
sudo systemctl restart ssh.socket
sudo systemctl restart ssh
```
> [!IMPORTANT]
> Verify your connection from Host-C (Kali) in a separate terminal before closing the current session:
> `ssh -p 22022 username@<Host-B-IP>`

## Phase 3: Deploying Dockerized Cowrie Honeypot (Host-B)

With port 22 free, Cowrie can now capture incoming traffic on standard ports.

### 1. Host-B Container Prep
```bash
sudo apt update && sudo apt install docker.io docker-compose-v2 -y
sudo usermod -aG docker $USER
newgrp docker # Refreshes user groups immediately
```

### 2. Initialize Directories
Create directories where Cowrie configurations and telemetry logs will live:
```bash
sudo mkdir -p /opt/cowrie/etc
sudo mkdir -p /var/log/cowrie
sudo chmod 777 /var/log/cowrie
```

### 3. Write Compose File
```bash
sudo nano /opt/cowrie/docker-compose.yml
```
Paste the following YAML configuration:
```yaml
version: '3'
services:
  cowrie-honeypot:
    image: cowrie/cowrie:latest
    container_name: cowrie-honeypot
    restart: always
    ports:
      - "22:2222"
      - "23:2223"
    volumes:
      - /var/log/cowrie:/cowrie/cowrie-git/var/log/cowrie
      - /opt/cowrie/etc:/cowrie/cowrie-git/etc
```

### 4. Start Cowrie Container
```bash
cd /opt/cowrie
docker compose up -d
```

## Phase 4: Deploying & Configuring Wazuh Agent (Host-B)

The local Wazuh Agent monitors logs generated by Cowrie and forwards them to Host-A.

### 1. Download and Install the Agent
Download the installer based on your host version (e.g., Ubuntu DEB x86-64):
```bash
wget https://packages.wazuh.com/4.x/apt/pool/main/w/wazuh-agent/wazuh-agent_4.7.2-1_amd64.deb
sudo WAZUH_MANAGER='<Host-A-IP>' WAZUH_AGENT_NAME='Honeypot-HostB' dpkg -i wazuh-agent_4.7.2-1_amd64.deb
```

### 2. Configure Agent Logging Matrix
Add Cowrie's local JSON output as a monitored source inside the Wazuh agent configuration:
```bash
sudo nano /var/ossec/etc/ossec.conf
```
Ensure the `<client>` IP addresses map correctly, then scroll to the bottom of the file and paste this block directly above the final `</ossec_config>` tag:
```xml
  <localfile>
    <log_format>json</log_format>
    <location>/var/log/cowrie/cowrie.json</location>
  </localfile>
</ossec_config>
```

### 3. Grant Permissions and Start the Agent Service
The `wazuh` user must have file-read privileges to access `/var/log/cowrie/cowrie.json`.
```bash
sudo chmod -R 755 /var/log/cowrie

# Reload configurations and start agent
sudo systemctl daemon-reload
sudo systemctl enable wazuh-agent
sudo systemctl restart wazuh-agent
```
Verify the connection is online and green:
```bash
sudo systemctl status wazuh-agent
```

## Phase 5: Configuring Custom Decoders & Rules (Host-A)

Wazuh Manager requires custom rules to process Cowrie-formatted logs.

### 1. Create Rules File on Host-A
```bash
sudo nano /opt/wazuh-docker/single-node/custom_cowrie_rules.xml
```
Paste this XML ruleset:
```xml
<group name="cowrie,honeypot,">
  <!-- Parent Rule: Matches all events with the cowrie. event ID prefix -->
  <rule id="100100" level="0">
    <decoded_as>json</decoded_as>
    <field name="eventid">^cowrie\.</field>
    <description>Cowrie honeypot grouping alerts.</description>
  </rule>

  <!-- SSH/Telnet Failed Logins -->
  <rule id="100101" level="5">
    <if_sid>100100</if_sid>
    <field name="eventid">cowrie.login.failed</field>
    <description>Honeypot Attack: Unauthorized SSH/Telnet Login Attempt Failed</description>
    <group>authentication_failed,behavior_unusual,</group>
  </rule>

  <!-- SSH/Telnet Successful Logins -->
  <rule id="100102" level="7">
    <if_sid>100100</if_sid>
    <field name="eventid">cowrie.login.success</field>
    <description>Honeypot Attack: Attacker successfully logged into fake environment</description>
    <group>authentication_success,compromise,</group>
  </rule>

  <!-- Commands Executed Inside the Decoy Shell -->
  <rule id="100103" level="6">
    <if_sid>100100</if_sid>
    <field name="eventid">cowrie.command.input</field>
    <description>Honeypot Alert: Attacker executed command in fake shell</description>
    <group>recon,invalid_command,</group>
  </rule>
</group>
```

### 2. Inject Configuration to Container
Move the configuration into the manager container and reload the service:
```bash
# Copy XML to container
sudo docker cp /opt/wazuh-docker/single-node/custom_cowrie_rules.xml single-node-wazuh.manager-1:/var/ossec/etc/rules/local_rules.xml

# Restart manager
sudo docker restart single-node-wazuh.manager-1
```
Now, Host-A is equipped to decode, index, and alert on Cowrie security incidents!
